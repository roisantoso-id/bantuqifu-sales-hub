## 一、 项目经理 (PM) 视角规划

PM 的核心诉求是：**“接盘”商机、拆解任务、调度资源、把控进度、验收成果。** 他的数据视野是整个**部门 (Department)**。

### 1. 对应的页面结构

| 页面名称 | 路由规划 (示例) | 页面形态 & 视图 |
| :--- | :--- | :--- |
| **交付大盘** | `/delivery/dashboard` | **数据图表视图**：展示本部门进行中的项目数、即将逾期的风险项目、部门成员负载率 (谁闲谁忙)。 |
| **项目大厅** | `/delivery/projects` | **数据表格 (Data Table)**：展示本部门所有从商机转化来的项目列表。支持按状态、客户名、负责人进行 URL 状态筛选。 |
| **项目详情与拆解台** | `/delivery/projects/[id]` | **分栏视图**：顶部是项目全局信息（截止日、客户信息）；下方是 **WBS (工作分解) 任务列表**。 |

### 2. 核心功能模块 (Features)

* **项目接管与认领：** 商机赢单流入后，PM 可以在“项目大厅”将无主项目指派给自己或同部门的其他 PM。
* **任务拆解与指派 (核心)：** 在项目详情页，将大项目拆分为多个具体的 `ServiceTask`。选择同部门的交付人员进行指派，并设定单项任务的“截止日期”和“提成/绩效基数”。
* **全局进度监控：** 实时查看项目下各个任务的流转状态，系统根据任务完成比例自动计算整个项目的进度条。
* **异常督办 (一键催办)：** 发现某个任务进度停滞，点击“催办”，系统自动给对应的交付人员发送强提醒。
* **成果验收与结算 (流转终点)：** 交付人员提交任务后，PM 负责审核。审核通过（验收），系统后台自动触发该任务的提成/绩效结算逻辑。

## 二、 交付执行人员视角规划

交付人员的核心诉求是：**明确今天干什么、便捷地提交工作成果、知道自己能拿多少钱。** 他的数据视野仅限**被指派给自己的任务**。

### 1. 对应的页面结构

| 页面名称 | 路由规划 (示例) | 页面形态 & 视图 |
| :--- | :--- | :--- |
| **我的工作台** | `/delivery/tasks` | **看板视图 (Kanban) 或 列表**：按状态列出自己的任务（待处理 / 进行中 / 待验收 / 已完成）。 |
| **任务执行面板** | `/delivery/tasks/[id]` | **左右分栏结构**：左侧为任务详情、客户要求、状态切换按钮；右侧为 **交付留痕时间轴 (Timeline)**。 |
| **我的收益/绩效** | `/delivery/commissions` | **账单列表**：展示个人已完成任务所产生的提成/绩效点数，以及结算状态。 |

### 2. 核心功能模块 (Features)

* **任务接收与状态流转：** 在工作台查收新任务，点击“开始处理”将状态变更为 `IN_PROGRESS`，完成后点击“提交验收”变更为 `PENDING_REVIEW`。
* **交付留痕与附件上传 (核心)：** 在任务执行面板的 Timeline 中，每一次操作（发评论说明进度、上传设计稿/代码链接/验收单）都会作为不可篡改的日志记录下来。
* **催办响应：** 收到 PM 的催办消息后，可以在对应的任务节点下快速回复当前卡点或进度。
* **收益对账：** 在绩效页面查看自己本月“干了多少活、预计能拿多少钱”，与财务/HR 的结算数据对齐。

## 三、 模块与数据流向总结

这套规划本质上是一个 **漏斗与闭环**：
1. **Sales Hub (上游) -> PM 大厅：** 商机转化为项目。
2. **PM 大厅 -> 拆解台 -> 执行工作台：** 项目裂变为任务，分发到人。
3. **执行工作台 -> Timeline留痕 -> PM 验收：** 任务转化为交付成果，向上汇报。
4. **PM 验收 -> 结算中心 (下游)：** 成果转化为提成，业务闭环。
收到！既然明确了以 `roisantoso-id/bantuqifu-sales-hub` 这个仓库的代码架构为绝对基准，我们的规划就必须严格贴合它现有的技术栈：**Next.js (App Router)、shadcn/ui 组件库、基于 URL 的状态管理 (searchParams)、以及基于部门的 RBAC**。

为了让你和开发人员（或 AI）能无缝地将代码合并进去，我把之前的规划转化为**精确到 Next.js 目录结构和 shadcn/ui 组件库**的工程级规划。

### 一、 项目经理 (PM) 模块详细规划

PM 的核心交互都在 `/delivery/projects` 目录下。

| 功能模块 | 页面路由 | 核心 UI 组件 (`shadcn/ui`) | 关键技术实现与交互 |
| :--- | :--- | :--- | :--- |
| **部门项目大厅** | `/projects` | `DataTable`, `Tabs`, `Badge` | 1. **URL 状态管理**：顶部使用 `Tabs` 切换项目状态（全部、进行中、已完成），点击时更新 URL 的 `?status=xxx`。<br>2. **数据隔离**：Server Component 中严格通过 `departmentId` 过滤数据。 |
| **项目指派/认领** | `/projects` | `DropdownMenu`, `toast` | 在表格的“操作”列，点击后可将项目的 `pmId` 修改为自己或同部门的其他 PM，触发 Server Action 后刷新页面。 |
| **WBS 任务拆解** | `/projects/[id]` | `Card`, `Dialog`, `Form` | 1. 项目详情页下方展示任务列表。<br>2. 点击“新建任务”弹出 `Dialog`，表单包含：任务名、截止日期、提成金额、选择执行人（仅限同部门）。使用 `react-hook-form` + `zod` 校验。 |
| **一键催办/督办** | `/projects/[id]` | `Button`, `Alert` | 针对进度滞后的子任务，点击“催办”按钮，触发 Server Action 在数据库写入一条 `NUDGE` 类型的记录，并给执行人发送系统通知。 |
| **成果验收** | `/projects/[id]` | `Sheet` (侧滑面板) | 点击“审核”，右侧滑出 `Sheet` 查看执行人提交的凭证（文件/链接）。点击“通过”后，任务变更为 `COMPLETED`，并触发提成生成。 |

### 二、 交付执行人员模块详细规划

执行人员的核心交互都在 `/delivery/tasks` 目录下，注重操作的便捷性和留痕的严谨性。

| 功能模块 | 页面路由 | 核心 UI 组件 (`shadcn/ui`) | 关键技术实现与交互 |
| :--- | :--- | :--- | :--- |
| **我的工作台** | `/tasks` | `Tabs`, `Card` 或 `DataTable` | 1. **数据隔离**：仅查询 `executorId` 为当前用户的任务。<br>2. **视图展示**：使用 `Tabs` 根据 `?status=` 过滤（待办 TODO、进行中 IN\_PROGRESS 等）。 |
| **状态流转引擎** | `/tasks/[id]` | `Select`, `Button` (带 loading) | 在任务详情页顶部，执行人员可以推进状态（例如：点击“开始执行”更新状态为 `IN_PROGRESS`，点击“提交验收”更新状态为 `PENDING_REVIEW`）。 |
| **交付留痕 (Timeline)** | `/tasks/[id]` | 自定义 Timeline UI, `Textarea` | 1. 类似 GitHub Issue 的交互。核心组件是一个输入框和上传按钮（对接 Supabase Storage）。<br>2. 提交后写入 `DeliveryRecord` 表，下方以时间轴形式渲染所有的操作日志、评论和附件。 |
| **绩效对账看板** | `/commissions` | `DataTable`, `Card` (统计数据) | 汇总该员工名下所有状态为 `APPROVED` 和 `SETTLED` 的提成记录。顶部用几个大 `Card` 展示“本月预计收益”、“累计已结算”。 |
